# 头条号与一点号提交证据修复设计

## 背景

2026-07-15 的受控真实测试证明：头条号和一点号均完成标题、正文及发布设置填写，代码也打印了“点击发布”，但页面仍停留在编辑器，正式列表和草稿列表均没有目标文章。现有实现仍返回 `submitted_unverified`，把“点击调用没有抛异常”错误地当成“平台可能已经接收提交”。

## 根因

`click_publish_common()` 存在四个缺口：

1. `HumanBehavior.human_click()` 在元素有边界框时使用坐标点击。点击被遮罩、元素状态变化或坐标落点异常时，调用仍可能正常返回。
2. 点击后的 30 秒等待没有成功条件断言。URL、反馈、弹窗、按钮状态全部不变时，函数仍正常返回。
3. 头条号确认文本包含泛化的“确定”，可能匹配与发布无关的按钮。
4. 一点号提交前没有验证发布按钮处于可交互状态。

结果验证本身能避免把管理页导航文字误判为成功，但它无法区分“平台可能接收了提交”和“主按钮根本没有生效”。

## 目标

- 发布按钮静默失效时立即阻断，并给出明确错误。
- 只有存在可观察的提交过渡，才允许进入后台结果核验。
- 只有正式列表精确标题或正式文章 URL 能证明 `published`。
- 草稿列表精确标题只能证明 `draft_only`。
- 不改变知乎、简书、B站及微信公众号行为。
- 不通过再次真实发表同一篇文章验证代码。

## 方案

头条号和一点号使用平台专用提交方法，不修改其他平台继续使用的通用点击路径。

### 提交前门禁

- 精确定位平台主发布按钮。
- 要求按钮存在、可见、启用。
- 记录点击前 URL、按钮可见状态及当前反馈文本。
- 头条号先关闭已知 AI 助手遮罩。

### 点击行为

- 使用 Playwright `Locator.click()`，让框架执行可见性、稳定性、遮挡和可交互检查。
- 不使用裸鼠标坐标点击。
- 头条号确认按钮只在可见发布对话框内查找，并只接受“确定并发布”或“确认发布”。删除泛化的“确定”。
- 一点号仅在平台真实出现发布确认框时点击确认；没有确认框不重复点击主按钮。

### 可观察提交证据

主按钮点击后，必须在限定时间内出现至少一种变化：

- URL 离开编辑器；
- 发布确认对话框出现；
- 明确且新出现的成功、失败或限流反馈；
- 主发布按钮消失或变为不可交互。

确认按钮点击后也必须出现 URL 变化、对话框关闭、新反馈或发布按钮状态变化。若没有任何变化，抛出 `publish_click_no_effect`，状态进入失败路径，不记录 `submitted_unverified`。

### 最终结果

- 正式文章 URL或正式管理列表精确标题：`published`。
- 草稿管理列表精确标题：`draft_only`。
- 已观察到真实提交过渡，但管理页暂时无法核验：保留 `submitted_unverified`，禁止自动重投。
- 没有观察到提交过渡：失败，`error_type=publish_click_no_effect`。
- 明确限流反馈：`limit_reached`，保留平台原始原因。

## 测试设计

先写失败测试并确认 RED：

1. 主按钮坐标点击静默无效时必须失败。
2. 禁用按钮必须在提交前失败。
3. 头条号不得把普通“确定”按钮当作发布确认。
4. 主按钮产生确认弹窗、确认按钮产生页面过渡时允许进入结果核验。
5. 一点号无确认框但 URL 或按钮状态发生变化时允许进入结果核验。
6. 无点击效果不能落为 `submitted_unverified`。
7. 正式列表、草稿列表及限流的现有测试继续通过。

完成定向测试后运行全量测试。自动化测试不访问真实平台。

## 真实验收边界

代码提交后不自动真实发表。真实验收前由用户指定一篇未发表文章；每个平台最多提交一次。平台后台精确标题或正式 URL 是最终验收证据。若未落库，停止重试并继续采样诊断信息。

## 非目标

- 不恢复五个平台封面上传。
- 不修改浏览器生命周期、隔离 profile 或无 CDP 架构。
- 不重构通用发布状态机。
- 不清理历史发布记录或人工修改状态文件。
